All Novel Can Be Galgame：AI 能力分层与模型路由方案 v1
一、总原则
系统采用：

规则优先 + 强模型兜底 + 小模型后续替代

更具体地说：

原则 1：不是所有任务都交给大模型
能用规则稳定解决的，不用大模型。
能用规则 + 轻量模型完成候选生成的，不直接让大模型从零思考。

原则 2：强模型 API 只用于高语义复杂度任务
尤其是：

归因
scene 切分
VN 映射
保真审查
原则 3：系统必须从一开始支持能力替换
今天是 GPT / Claude，明天可能是本地模型或混合模型。
所以设计上要面向“能力接口”，而不是绑定某个厂商。

原则 4：本地工作台负责流程、缓存、调度与状态
大模型负责复杂理解与生成。
本地不等于全离线，而是本地掌控工作流与数据。

原则 5：未来小模型只替代高频、低风险、模式稳定的任务
不要一开始就想让小模型接管最难的任务。

二、能力分层总览
建议把 AI 能力分成 4 层：

text
 
L0: 规则层
L1: 轻量模型层（未来可选）
L2: 强模型 API 层
L3: 审核与路由层
L0：规则层
特点：

本地执行
成本极低
可解释
稳定
适合格式化、清洗、初筛、归一化
L1：轻量模型层
特点：

本地或低成本部署
适合分类、抽取、候选生成
不承担最终复杂推理责任
L2：强模型 API 层
特点：

负责复杂语义理解、推理、映射、审查
成本高
效果最好
当前阶段是主力
L3：审核与路由层
特点：

决定某个任务应该走哪层
负责 fallback
负责预算控制
负责缓存命中
负责失败重试

四、每个任务的能力分层建议
任务 A：文本清洗
当前推荐层：L0 规则层

原因：

编码、空白、平台尾巴、固定噪声模式，都是规则问题
是否需要强模型：

通常不需要
是否适合小模型：

不优先，规则足够
路由策略：

默认规则执行
仅在异常文本时人工标记或后续补专门清洗器
任务 B：结构识别
当前推荐层：L0 规则层为主，L2 强模型兜底

规则可处理内容：

第X章
第一章
Chapter X
番外
后记
卷一/第一卷
强模型介入场景：

标题格式极不规范
章节边界模糊
目录和正文错位
结构混乱文本
未来是否适合小模型：

适合，但优先级不高
路由策略：

先规则识别
若结构置信度低，再走强模型修正
结果进入一致性检查
任务 C：叙事单元解析
当前推荐层：L2 强模型 API 为主，L0 规则辅助

原因：

中文恋爱小说里心理描写和旁白混杂
动作描写常嵌在叙事中
对话外包裹叙述很多
规则可做什么：

引号中的内容初判为 dialogue
某些心理词、动作词做弱提示
段落切分
强模型负责什么：

最终判断每个单元类别
处理混合句
保持上下文理解
未来小模型潜力：

很高
是最适合后续训练小模型的任务之一
路由策略：

规则先做初步切分和候选标签
强模型做最终解析
后续可替换成“小模型初判 + 强模型难例兜底”
任务 D：归因
当前推荐层：L2 强模型 API 为主，L0/L1 提供候选

原因：

中文小说里说话人归因非常难
连续对话不写名字
主语省略
代词模糊
心理与外在说话交错
规则可做什么：

最近出现角色候选
引号前后的“某某说”
代词和名字近邻匹配
当前 scene 参与者候选
强模型负责什么：

在上下文中判断 speaker / actor
处理模糊归因
输出 uncertain 标记
未来小模型潜力：

中高
适合做候选生成，不适合一开始做最终裁决
路由策略：

规则生成候选角色列表
强模型做最终归因
若置信度低，标记 uncertain 并交给 review
任务 E：scene 切分
当前推荐层：L2 强模型 API 为主，L0 规则辅助

原因：

scene 边界不是纯格式问题，而是叙事问题
规则可做什么：

地点词变化提示
时间词变化提示
段落大跳跃提示
回忆/插叙提示词
强模型负责什么：

判断 scene 是否真的应切开
给出边界原因
保持 VN 节奏合理
未来小模型潜力：

高
适合训练边界检测模型
路由策略：

规则先标出候选边界
强模型确认和修正
后续可用小模型做边界预测，强模型只审难例
任务 F：VN 映射
当前推荐层：L2 强模型 API

原因：

这是当前系统最复杂的语义转译任务
它要做忠实映射、舞台化、节奏控制、step 类型选择
规则可做什么：

输出格式校验
step schema 校验
简单模板拼接（极少）
未来小模型潜力：

低到中
前期不建议替代
路由策略：

直接走强模型
前置输入由上游 Agent 结构化后再送入
输出必须经过 schema 校验和 fidelity review
任务 G：保真审查
当前推荐层：L2 强模型 API + L0 规则校验

原因：

保真审查需要比较原文、归因结果、VN 脚本并判断是否偏离
这是高阶审查任务
规则可做什么：

原文台词匹配率
顺序一致性初检
非原文新增文本检测
step 合法性检测
强模型负责什么：

语义级保真判断
是否有隐性改写
是否有不当补充
是否漏掉关键内容
未来小模型潜力：

低
前期不建议
路由策略：

规则先做硬性检查
强模型做语义保真审查
不通过则回退 VN Mapping 重做或人工确认
任务 H：视觉 prompt 提取
当前推荐层：L0 + L2 混合

原因：

一部分很适合规则抽取：时间、天气、地点、外貌关键词、服装关键词
但要整理成高质量 prompt，强模型更合适
规则可做什么：

关键词抽取
角色外貌片段收集
场景描述片段收集
氛围词收集
强模型负责什么：

将证据整理成 prompt
区分“原文明确”和“保守补全”
控制风格模板
未来小模型潜力：

高
很适合做信息抽取小模型
路由策略：

规则/小模型抽视觉证据
强模型整理为最终 prompt
gpt-image-2 生成图像
任务 I：一致性检查
当前推荐层：L0 规则层为主，L2 强模型补充

原因：

很多一致性问题是工程问题：
命名不一致
标签不一致
背景 id 不一致
角色别名未归一
规则可做什么：

字符串归一
别名表
标签表
背景复用检测
强模型介入场景：

某些语义近似标签是否应合并
场景语义是否重复
未来小模型潜力：

中等，但不急
路由策略：

默认规则检查
复杂语义冲突再走强模型

六、模型路由策略
建议采用三段式路由：

text
 
预判 → 执行 → 审核/回退
预判阶段
系统先判断：

任务类型
输入复杂度
规则是否足够
是否已有缓存
当前预算是否允许走强模型
是否属于高风险任务
执行阶段
路由优先级：

先查缓存
再走规则
再走小模型（如果有）
最后走强模型 API
但不是所有任务都要经过所有层。
例如 VN Mapping 可以直接走强模型。

审核/回退阶段
执行完后做：

schema 校验
一致性检查
fidelity review
失败重试或降级
七、置信度与升级机制
建议每个任务都输出一个简化置信度：

high
medium
low
示例：结构识别
如果规则识别到：

标题格式统一
边界清晰
章节数合理
则 high，否则 low，升级强模型。

示例：Attribution
如果：

明确有“林秋说”
当前 scene 只有两人
代词清晰
则 medium/high

如果连续三句对话都无名字、多人在场，则 low，需要强模型精判。

示例：Visual Prompt
如果原文明确出现：

黑长发
蓝色校服
雨夜便利店
则 high

如果外貌描述极少，则 low，只能做保守 prompt。

八、预算路由策略
建议支持 3 种模式：

1. 高质量模式
高复杂度任务全部走强模型
中复杂度任务也尽量走强模型
适合最终导出
2. 平衡模式
规则优先
中复杂度任务尽量混合
高复杂度任务走强模型
适合日常工作
3. 省钱模式
能规则就规则
中复杂度任务尽量走小模型/简化策略
强模型只用于关键章节或关键任务
适合初筛和批处理
九、缓存策略
建议缓存分 4 层：

1. 原始清洗缓存
同一本书清洗一次即可。

2. 章节解析缓存
章节结构、叙事单元、归因结果都缓存。

3. scene 缓存
scene 切分和 scene 映射按章缓存。

4. prompt 缓存
角色 prompt、背景 prompt 缓存。

缓存失效规则
改视觉风格，不应让文本解析缓存失效
改 scene 切分策略，只影响切分和后续映射
改 VN 映射策略，不应影响结构识别
十、失败与回退策略
结构识别失败
回退到人工确认章节边界
或标记该书不进入核心流程
叙事单元解析失败
降级为更粗粒度段落解析
或只保留 dialogue/narration 两类
Attribution 失败
标记 uncertain
在播放层用“???”或旁白包裹
后续 review 再修
VN Mapping 失败
重试一次
再不行降级成“保守映射模式”
少表情
少舞台指令
多保留原文旁白
Fidelity Review 不通过
回退 VN Mapping 重做
或进入人工确认

结合当前前提：

产品定位：本地可部署工作台
AI 主力：外部强模型 API
未来可引入：小模型
改编原则：忠实转译，不改剧情事实
视觉层：先简化为 prompt 提取器 + gpt-image-2
数据类型：中文恋爱向 txt 小说
v1 的 Agent 体系正式定为：

核心生产 Agent
Structure Agent
Narrative Parsing Agent
Attribution Agent
Scene Segmentation Agent
VN Mapping Agent
Visual Prompt Agent
核心审查 Agent
Fidelity Review Agent
Consistency Review Agent
核心调度 Agent
Orchestrator Agent
二、Agent 分层定位
第一层：内容理解层
Structure Agent
Narrative Parsing Agent
Attribution Agent
第二层：VN 转译层
Scene Segmentation Agent
VN Mapping Agent
Visual Prompt Agent
第三层：质量控制与流程层
Fidelity Review Agent
Consistency Review Agent
Orchestrator Agent
三、总设计原则
原则 1：Agent 只负责一个清晰任务域
不要让一个 Agent 同时做：

解析
改写
审查
视觉规划
原则 2：Agent 输出必须结构化
每个 Agent 的输出都应该是：

可缓存
可检查
可重跑
可供下游消费
原则 3：Agent 默认不改原文事实
除 VN Mapping 的舞台化映射外，其他 Agent 都不应主动改写原文内容。

原则 4：审查 Agent 不负责创作
Review Agent 负责：

找问题
给建议
输出 patch 提示
但不直接承担上游主任务。

原则 5：Orchestrator 负责流程，不负责内容判断
总控 Agent 不应自己顺手改内容，它只负责：

调度
状态流转
重试
路由
回退
原则 6：每个 Agent 都要有禁止行为
边界清晰不仅要定义“做什么”，还要定义“不能做什么”。

番外/后记/特殊章节识别
章节正文切分
目录树构建（如果可识别）

不负责角色识别
不负责对话归因
不负责 scene 切分
不负责 VN 脚本生成
不负责视觉 prompt
不负责剧情理解

引号内外关系

对 Narrative Units 做角色归属。

输出不确定归因标记

不负责角色完整人设生成
不负责 VN 脚本映射
不负责视觉设计
不负责改写台词
不负责 scene 切分最终决策

当前主力：L2 强模型 API
辅助：L0 规则候选
未来方向：L1 候选生成模型
输入
Annotated Narrative Units

输出
Attributed Narrative Units
Character Mention Map
Speaker/Actor/Thinker 归因结果

别名归一结果
成功标准
对话说话人归属准确

七、Scene Segmentation Agent
目标
在不改变原文顺序和事实的前提下，把章节切成多个 scene。

负责内容
识别 scene 边界
标记边界原因：
地点变化
时间变化
叙事焦点变化

十六、最关键的边界冲突提醒
1. Narrative Parsing Agent vs Attribution Agent
Parsing 只标类型，不判归属。
归属必须由 Attribution Agent 负责。

2. Scene Segmentation Agent vs VN Mapping Agent
scene 边界由 Scene Segmentation Agent 决定。
VN Mapping 只能消费 scene，不应私自改 scene。

3. Visual Prompt Agent vs VN Mapping Agent
视觉层只能基于文本与 scene 结果提取，不得反向改剧情。

4. Fidelity Review Agent vs VN Mapping Agent
Review 只指出问题和 patch 建议，不负责主生成。

5. Orchestrator vs 所有内容 Agent
Orchestrator 只调度，不写内容。
