一、MVP 一句话定义
一个本地可部署的工作台，支持导入中文恋爱向 txt 小说，自动完成章节解析、叙事解析、角色归因、scene 切分、VN 脚本映射、保真审查，并生成可预览的视觉小说 Demo；视觉层先采用 prompt 提取 + gpt-image-2。

二、MVP 的产品目标
MVP 不是为了“一步到位做成完整商业 galgame 生产平台”，而是为了验证三件事：

1. 技术可行性
证明：

中文恋爱向 txt 小说可以较稳定地转成 VN 结构
核心 Agent 工作流可以跑通
忠实转译是可实现的
2. 产品价值
证明：

用户愿意把喜欢的小说转成可玩的视觉小说
本地工作台 + 外部模型 API 的形态可接受
分章预览是有吸引力的
3. 工程闭环
证明：

Agent 可评测
工作流可缓存
结果可预览
失败可回退
后续可扩展
三、MVP 的范围定义
3.1 MVP 必做范围
A. 输入能力
导入 txt 小说
支持中文文本
支持带章节标题的长篇小说
支持基础参数配置：
忠实度模式
scene 切分策略
视觉风格模板
预算模式
B. 核心文本处理链路
必须包含：

Structure Agent
Narrative Parsing Agent
Attribution Agent
Scene Segmentation Agent
VN Mapping Agent
Fidelity Review Agent
C. 轻量视觉链路
Visual Prompt Agent
输出角色 prompt / 背景 prompt
接 gpt-image-2
图片生成失败不阻塞文本主链路
D. 工作流能力
章节驱动处理
scene 级缓存
失败重试
保守模式 fallback
局部重跑
分章预览
E. 预览能力
Web 预览播放器
支持：
narration
say
thought
background
show/hide
basic transition
支持按章节预览
F. 本地工作台能力
本地项目管理
API Key 配置
模型提供商配置
任务执行状态查看
缓存与结果查看
3.2 MVP 可选但非必须
角色立绘图像实际生成
背景图像实际生成
CG prompt
多模型 provider 切换
复杂一致性审查
批量项目管理
导出 Ren'Py
3.3 MVP 明确不做
不做 1：剧情分支
不做 2：用户修改原文内容
不做 3：复杂美术系统
不做 4：全本地 AI
不做 5：多人协作和社区

五、MVP 的成功标准
5.1 技术成功标准
70% 以上的目标小说能够完成结构解析
85% 以上的章节可跑到 chapter_ready
90% 以上的 scene 可生成可播放脚本
Fidelity Review 能有效拦截明显偏离
5.2 产品成功标准
用户可以：

导入一本恋爱向 txt 小说
在合理时间内看到前几章 playable demo
感受到“这确实像 galgame”
同时又觉得“没有乱改原文”
5.3 工程成功标准
全流程有缓存
支持分章重跑
失败不会导致全项目报废
本地工作台可稳定使用
API 成本可控
六、MVP 的最小交付形态
上传一本带章节的中文恋爱向 txt 小说，系统可自动生成前 1~3 章的可玩 VN 预览。

不是整本书全部完成，而是：

先把前若干章稳定跑通
证明工作流和体验成立
七、里程碑拆解总览
建议拆成 5 个阶段：

Phase 1：文本主链路打通
Phase 2：工作流与本地工作台

十五、MVP 风险清单
风险 1：Parsing 和 Attribution 不稳
对策：

先把 Gold Set 做好
优先打磨这两个 Agent
不要过早追求视觉效果
风险 2：Scene 切分不好，导致 VN 感差
对策：

单独做 scene 评测
保守切分优先
不要过度戏剧化
风险 3：VN Mapping 偷偷改写原文
对策：

Fidelity Review 必须强
默认保守映射模式
严控新增文本率
风险 4：本地工作台流程太重
对策：

MVP 交互尽量简单
先只支持 txt
默认配置尽量少
风险 5：图像生成拖慢整体体验
对策：

视觉链路不阻塞主链路
先文本 playable，再补图
