一、总体架构建议
我建议项目整体分成 5 层：

text
 
apps/            # 可运行应用
packages/        # 共享包
services/        # 本地后端服务/任务执行
data/            # 本地项目数据与缓存
docs/            # 产品/技术文档
更具体一点：

apps/workbench：本地工作台前端
apps/api：本地 API / orchestration 服务
packages/core：领域模型、schema、类型定义
packages/agents：各 Agent 实现
packages/runtime：VN 播放运行时
packages/providers：模型 API 适配层
packages/storage：本地存储与缓存层
packages/evaluation：评测与回归工具
data/projects：用户项目数据
data/cache：模型与中间结果缓存

三、目录结构背后的设计原则
1. apps 和 packages 分离
这样可以把：

前端工作台
本地 API 服务
Agent 引擎
runtime
provider 适配
拆开，后面更容易维护。

2. prompts 独立成包
这点我很建议你一开始就这么做。
因为你后面会频繁改：

Narrative Parsing prompt
Attribution prompt
VN Mapping prompt
Fidelity Review prompt
如果 prompt 散落在代码里，后面很难管理版本。

3. data 独立为本地数据根目录
因为你是本地工作台，所以项目数据、缓存、日志、生成结果都应该落盘。

4. packages/core 统一定义 schema 和状态
这样前后端、Agent、Orchestrator 都共用同一套类型和 schema，避免混乱。

四、项目内数据目录建议
每个项目建议单独一个目录：

text
 
data/projects/<project_id>/
├─ project.json
├─ raw/
├─ normalized/
├─ chapters/
├─ scenes/
├─ scripts/
├─ prompts/
├─ reports/
├─ preview/
└─ logs/
下面是每一层的作用。

4.1 project.json
保存项目元信息和全局状态。

4.2 raw/
保存原始输入文件。

例如：

text
 
raw/
└─ novel.txt
4.3 normalized/
保存清洗后的文本和结构结果。

例如：

text
 
normalized/
├─ cleaned.txt
├─ structure.json
└─ chapter_index.json
4.4 chapters/
按章节保存中间结果。

例如：

text
 
chapters/
├─ chapter-0001/
│  ├─ source.txt
│  ├─ narrative_units.json
│  ├─ attributed_units.json
│  ├─ segmentation.json
│  └─ chapter_state.json
├─ chapter-0002/
│  └─ ...
4.5 scenes/
按 scene 保存结果。

例如：

text
 
scenes/
├─ scene-0001/
│  ├─ scene.json
│  ├─ mapping_input.json
│  ├─ vn_script.json
│  ├─ fidelity_report.json
│  ├─ visual_prompt.json
│  └─ scene_state.json
4.6 scripts/
保存按章节或全书聚合后的脚本。

例如：

text
 
scripts/
├─ chapter-0001-script.json
├─ chapter-0002-script.json
└─ full-preview-script.json
4.7 prompts/
保存视觉 prompt 和可能的图像生成请求。

4.8 reports/
保存：

fidelity 汇总
consistency 汇总
项目处理报告
4.9 preview/
保存播放器直接消费的数据快照。

4.10 logs/
保存项目级日志和任务日志。

二十四、数据库还是文件？
你这个项目我建议 P0 采用“文件系统 + 轻量数据库索引”混合模式。

为什么不是纯数据库
因为你有很多：

原始文本
中间 JSON
prompt
script
report
预览快照
这些天然适合落文件。

为什么也不是纯文件
因为你还需要方便地查：

项目列表
任务状态
最近更新时间
哪些章节 ready
哪些 scene failed
这些更适合用轻量 DB 做索引。

推荐方案
文件系统：存正文和中间结果 JSON
SQLite：存项目、章节、scene、任务索引

二十九、最终结论
如果现在正式拍板，这份目录结构和数据结构的核心结论是：

项目应采用 monorepo 结构，将本地工作台前端、本地 API 服务、Agent 引擎、provider 适配、runtime、storage 和评测工具分层组织；数据层采用“文件系统存正文与中间结果 + SQLite 存索引与任务状态”的混合方案；所有 Agent 输出都以统一 schema 落盘，保证可缓存、可重跑、可评测。
